Highlights

Adds a native Swift port of the oref0 algorithm and plugs it into Trio as an opt‑in alternative to the existing JavaScript execution path. The port implements core algorithm modules in Swift (IoB, Meal/CoB, Autosens, DetermineBasal including Dynamic ISF), adds JSON bridging and conversion helpers, wiring in makeProfile to call both JS and Swift implementations for comparison, and includes tests and logging/replay support used to verify parity.

Opt‑in feature: the app’s default remains the JavaScript oref path; users can enable “Use Swift Oref” in Settings → Algorithm → Additionals to run the Swift implementation instead (a green “Swift Oref” badge is shown when enabled). The PR includes shadow‑execution and unit test verification aimed at parity between JS and Swift; enabling the Swift path will cause the app to use Swift‑generated algorithm outputs instead of the JS outputs. The change was developed with parity checks (including a large replay of inputs) but because this is a replacement of dosing algorithm logic, a conservative human review / validation for live dosing is recommended before broad deployment.

Adds a defensive minimum glucose floor in BaseGlucoseStorage: a new Config.minimumGlucose = 39 and a private clampToMinimum(_:) helper that maps BloodGlucose entries to ensure glucose and sgv are at least 39 (and emits a debug(.deviceManager, ...) message with the raw value and timestamp). storeGlucose(_) and backfillGlucose(_) run their inputs through clampToMinimum before filtering/persistence, and storeCGMState is called with the clamped array. addManualGlucose is intentionally left unchanged so user-entered fingerstick values are not rewritten. Unit tests verify sub-39 CGM readings are clamped on storeGlucose and backfillGlucose.

CGM-sourced glucose readings below 39 mg/dL will be clamped to 39 before being filtered or persisted, which may change how very-low CGM readings influence determinations or stored history when upstream plugins forward values below the vendor floor. User-entered manual/fingerstick glucose values are not modified by this change. The PR also logs the raw low CGM reading and timestamp for diagnostics rather than silently discarding it. If you need a precise assessment of how this affects closed-loop dosing in all deployment configurations, human review is recommended.

Replaces the previous fixed glucose fetchLimit (72 readings = 6h) with a dynamic calculation tied to preferences.maxMealAbsorptionTime. Preferences are loaded earlier and the glucose fetch now uses either a computed fetchLimit based on maxMealAbsorptionTime or a fetchHours = maxMealAbsorptionTime + 0.5 approach; fetchAndProcessGlucose was extended to accept fetchHours and the glucose query now uses a time cutoff computed from fetchHours hours ago.

Glucose history supplied to the meal/COB logic is now fetched to cover the configured maxMealAbsorptionTime (plus a 0.5h buffer in the final change) instead of being limited to 6 hours. The intent is to prevent the reported COB rise and cliff-drop when users set MMAT > 6h; the change is applied upstream of determineBasal so it affects both JS oref and the native oref-swift path. The PR includes test suggestions to verify COB trajectories and glucoseFetchLimit/hours scaling.
